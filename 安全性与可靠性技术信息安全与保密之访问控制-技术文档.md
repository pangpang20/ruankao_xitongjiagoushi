## 1. Overview

### 1.1 Definition | 定义

**Access Control** is a security mechanism that regulates who or what can view or use resources in a computing environment. It is a fundamental concept in security that minimizes risk to the business or organization by ensuring that only authorized users can access specific resources, data, or services.

**访问控制**是一种安全机制,用于规范谁或什么可以查看或使用计算环境中的资源。它是安全中的基本概念,通过确保只有授权用户可以访问特定资源、数据或服务来最小化业务或组织的风险。

### 1.3 Access Control Objectives | 访问控制目标

**Primary Objectives:**

**主要目标:**

1. **Confidentiality (机密性)** - Prevent unauthorized disclosure of information
   - 防止未经授权披露信息

2. **Integrity (完整性)** - Prevent unauthorized modification of data
   - 防止未经授权修改数据

3. **Availability (可用性)** - Ensure authorized users can access resources when needed
   - 确保授权用户在需要时可以访问资源

4. **Accountability (问责性)** - Track and audit user actions
   - 跟踪和审计用户操作

5. **Non-repudiation (不可否认性)** - Prevent denial of actions
   - 防止否认操作

### 2.3 Access Control Components | 访问控制组件

**Core Components:**

**核心组件:**

1. **Subject (主体)**
   - User, process, or system entity requesting access
   - 请求访问的用户、进程或系统实体
   - Examples: End user, service account, application, API client
   - 示例: 最终用户、服务账户、应用程序、API客户端

2. **Object (客体)**
   - Resource being accessed
   - 被访问的资源
   - Examples: File, database, network service, application, API endpoint
   - 示例: 文件、数据库、网络服务、应用程序、API端点

3. **Access Rights (访问权限)**
   - Permissions granted to subjects
   - 授予主体的权限
   - Common rights: Read, Write, Execute, Delete, Modify, Create
   - 常见权限: 读取、写入、执行、删除、修改、创建

4. **Reference Monitor (参考监视器)**
   - Mediates all access requests
   - 调解所有访问请求
   - Enforces access control policies
   - 执行访问控制策略

## 4. Authorization Models

### 4.1 Discretionary Access Control (DAC) | 自主访问控制

**Definition:**

Owner of a resource determines who can access it.

资源所有者决定谁可以访问它。

**Example: Unix/Linux File Permissions**

**示例: Unix/Linux文件权限**

```bash
# File permissions example
# 文件权限示例

-rwxr-xr--  1  alice  staff  4096  Dec 23 10:30  document.txt

┌─┬─────┬─────┬─────┐
│-│rwx  │r-x  │r--  │
└─┴─────┴─────┴─────┘
  │  │     │     │
  │  │     │     └─── Others: read only (其他用户:只读)
  │  │     └───────── Group: read, execute (组:读、执行)
  │  └─────────────── Owner: read, write, execute (所有者:读、写、执行)
  └────────────────── File type: regular file (文件类型:普通文件)

# Change permissions
# 更改权限
chmod 750 document.txt
chown alice:staff document.txt
```

**Strengths & Weaknesses:**

**优势与劣势:**

| Strengths 优势                 | Weaknesses 劣势              |
| ------------------------------ | ---------------------------- |
| Flexible and easy to implement | Difficult to manage at scale |
| 灵活且易于实施                 | 难以大规模管理               |
| User-friendly                  | Prone to privilege creep     |
| 用户友好                       | 容易出现权限蔓延             |
| Fine-grained control           | No centralized policy        |
| 细粒度控制                     | 无集中策略                   |

### 4.2 Mandatory Access Control (MAC) | 强制访问控制

**Definition:**

System enforces access control based on security labels and clearance levels.

系统根据安全标签和许可级别强制执行访问控制。

**Bell-LaPadula Model (Confidentiality Focus):**

**Bell-LaPadula模型(机密性焦点):**

```
Security Levels
安全级别

    ┌─────────────────────┐
    │   TOP SECRET        │  Level 4 (最高机密)
    ├─────────────────────┤
    │   SECRET            │  Level 3 (机密)
    ├─────────────────────┤
    │   CONFIDENTIAL      │  Level 2 (秘密)
    ├─────────────────────┤
    │   UNCLASSIFIED      │  Level 1 (非机密)
    └─────────────────────┘

Bell-LaPadula Rules:
Bell-LaPadula规则:

1. No Read Up (简单安全属性)
   - Subject at level N cannot READ objects at level > N
   - N级别主体不能读取>N级别的对象
   Example: SECRET user cannot read TOP SECRET documents
   示例: SECRET用户不能读取TOP SECRET文档

2. No Write Down (*-属性)
```

## 6. Privileged Access Management (PAM)

**Definition:**

Security solution for managing and monitoring privileged accounts with elevated permissions.

用于管理和监控具有提升权限的特权账户的安全解决方案。

**特权账户包括:**

   - AES-256 encryption

2. **Session Management (会话管理)**
   - Privileged session recording (video)
   - Real-time monitoring
   - Keystroke logging

3. **Privilege Elevation (特权提升)**
   - Just-In-Time (JIT) access
   - Time-limited privilege grants
   - Automatic privilege revocation

## 10. Attack Vectors and Countermeasures

### 10.1 Common Attacks | 常见攻击

**1. Credential Theft (凭证盗窃)**

- **Attack:** Phishing, keylogging, password dumps
- **Countermeasure:** MFA, passwordless authentication, anti-phishing training

**2. Privilege Escalation (特权提升)**

- **Attack:** Exploiting misconfigurations, vulnerable applications
- **Countermeasure:** Least privilege, PAM, regular access reviews

**3. Session Hijacking (会话劫持)**

- **Attack:** Session token theft, man-in-the-middle
- **Countermeasure:** Encrypted sessions (TLS), short session timeouts

**4. Brute Force (暴力破解)**

- **Attack:** Password guessing, credential stuffing
- **Countermeasure:** Account lockout, rate limiting, CAPTCHA

**5. Social Engineering (社会工程)**

- **Attack:** Impersonation, pretexting
- **Countermeasure:** Security awareness training, verification procedures

## 11. Implementation Best Practices

### 11.1 Security by Design | 设计安全

1. **Implement Least Privilege from Day One**
   - 从第一天就实施最小权限
   - Start with minimal permissions, grant additional access as needed

2. **Centralized Identity Management**
   - 集中式身份管理
   - Single source of truth for identities (Active Directory, Azure AD, Okta)

3. **Automated Provisioning and De-provisioning**
   - 自动化配置和取消配置
   - Integration with HR systems for joiner/mover/leaver processes

4. **Regular Access Reviews (Certifications)**
   - 定期访问审查(认证)
   - Quarterly or bi-annual review of user permissions
   - Manager attestation for direct reports

5. **Privileged Access Monitoring**
   - 特权访问监控
   - Real-time alerts for privileged operations
   - Session recording for forensic investigation

6. **Security Awareness Training**
   - 安全意识培训
   - Regular phishing simulations
   - Password hygiene education

### 11.2 Implementation Roadmap | 实施路线图

**Phase 1: Foundation (基础阶段)**
- Inventory all accounts and permissions
- Implement centralized IAM
- Enable MFA for all users

**Phase 2: Strengthen (加强阶段)**
- Deploy PAM solution for privileged accounts
- Implement RBAC model
- Establish access certification process

**Phase 3: Optimize (优化阶段)**
- Transition to ABAC for dynamic policies
- Implement Zero Trust architecture
- Enable continuous authentication

**Phase 4: Mature (成熟阶段)**
- AI-powered anomaly detection
- Automated policy recommendations
- Full SIEM integration

**访问控制的未来：**

- **AI-Powered Access Management:** Machine learning for anomaly detection and risk-based authentication
- **AI驱动的访问管理：** 用于异常检测和基于风险的身份验证的机器学习
- **Passwordless Future:** FIDO2, biometrics, and behavioral authentication
- **无密码未来：** FIDO2、生物识别和行为身份验证
- **Decentralized Identity:** Self-sovereign identity (SSI), blockchain-based identity
- **去中心化身份：** 自主身份(SSI)、基于区块链的身份
- **Continuous Adaptive Risk and Trust Assessment (CARTA):** Dynamic, context-aware access decisions
- **持续自适应风险和信任评估(CARTA)：** 动态、上下文感知的访问决策
