## Introduction | 概述

### What is Application System Design and Development? | 什么是应用系统设计与开发?

**English:**

**Application System Design and Development** is the systematic process of creating software applications that meet specific business or user needs. It encompasses the entire software development lifecycle (SDLC), from initial requirements analysis through design, implementation, testing, and deployment.

**Key Phases | 关键阶段**:
```
Requirements → Analysis → Design → Implementation → Testing → Deployment → Maintenance
需求收集    → 分析    → 设计  → 实现        → 测试   → 部署      → 维护

Focus of This Document | 本文档重点:
         Analysis & Design Methods | 分析与设计方法
                  ↓
         External Design | 外部设计
                  ↓
         Internal Design | 内部设计
                  ↓
         Programming | 程序设计
                  ↓
         Testing | 测试
```

**Core Principles | 核心原则**:
1. **User-Centric | 以用户为中心**: Design for end-user needs and experience
2. **Modularity | 模块化**: Decompose system into manageable components
3. **Abstraction | 抽象**: Hide complexity, expose essential features
4. **Quality-Driven | 质量驱动**: Build quality in from the start
5. **Iterative Improvement | 迭代改进**: Continuously refine through feedback

**中文:**

**应用系统设计与开发**是创建满足特定业务或用户需求的软件应用的系统化过程。它涵盖从初始需求分析到设计、实现、测试和部署的整个软件开发生命周期（SDLC）。

### Why Application System Design Matters | 应用系统设计的重要性

**English:**

**For Software Quality | 对软件质量**:
- **Maintainability**: Well-designed systems are easier to modify and extend
- **Reliability**: Proper design reduces defects and failures
- **Performance**: Good architecture ensures scalability and efficiency

- **Cost Reduction**: Good design reduces rework and maintenance costs
- **Time-to-Market**: Clear design accelerates development

- **Core Topic**: Design and testing account for 25-30% of exam content
- **Case Analysis**: Design trade-offs and testing strategies in scenarios
- **Method Selection**: Choosing appropriate analysis and design methods
- **Quality Assurance**: Understanding testing techniques and quality models

**中文:**

**对软件质量**:
- **可维护性**: 设计良好的系统更易于修改和扩展
- **可靠性**: 适当的设计减少缺陷和故障
- **性能**: 良好的架构确保可扩展性和效率
- **安全性**: 设计层面的安全考虑防止漏洞

**对项目成功**:
- **成本降低**: 良好的设计减少返工和维护成本
- **上市时间**: 清晰的设计加速开发

### 1.2 Structured Method | 结构化方法

**English:**

**Structured Method** is a software development approach that emphasizes top-down functional decomposition and process-oriented modeling.

**Three Components | 三个组成部分**:

#### 1.2.1 Structured Analysis (SA) | 结构化分析

**Purpose | 目的**: Model what the system does (functional requirements)

**Key Tools | 关键工具**:

**1. Data Flow Diagram (DFD) | 数据流图**

**DFD Elements | DFD元素**:
```
┌─────────────┐
│  External   │  Data Source/Sink
│  Entity     │  数据源/数据汇
│  外部实体   │
└─────────────┘

     ──→       Data Flow
              数据流

┌─────────────┐
│   Process   │  Transform data
│   处理      │  转换数据
└─────────────┘

┌─────────────┐
│ Data Store  │  Persistent data
│ 数据存储    │  持久化数据
└─────────────┘
```

**DFD Levels | DFD层次**:
```
Context Diagram (Level -1) | 上下文图:
┌────────┐         ┌────────────┐         ┌────────┐
│Customer│────────→│   System   │────────→│ Bank   │
│ 客户   │         │   系统     │         │ 银行   │
└────────┘         └────────────┘         └────────┘

Level 0 DFD | 0层图:
┌────────┐    Order    ┌──────────┐  Payment  ┌─────────┐
│Customer│────────────→│  Process │──────────→│ Payment │
│        │             │  Order   │           │ System  │
│        │←─Confirmation│  订单处理│           │ 支付系统│
└────────┘             └────┬─────┘           └─────────┘
                            │
                            ▼
                       ┌──────────┐
                       │ Order DB │
                       │ 订单数据库│
                       └──────────┘

Level 1 DFD (Decomposition of Process Order) | 1层图:
Detail of \"Process Order\" broken into sub-processes
\"订单处理\"细化为子流程
```

**2. Data Dictionary (DD) | 数据字典**

**Purpose**: Define all data elements, structures, and flows

**Example Entry | 示例条目**:
```
Data Element: Customer_Name
Name: Customer_Name
Type: String
Length: 50
Description: Full name of the customer
Format: Last Name, First Name
Example: Smith, John
Constraints: Not null, no special characters

Data Structure: Order
Order = Order_ID + Customer_ID + Order_Date + {Order_Item} + Total_Amount
Order_Item = Product_ID + Quantity + Unit_Price
```

**3. Process Specification (PSPEC) | 处理说明**

**Methods | 方法**:
- **Structured English | 结构化英语**: Pseudo-code description
- **Decision Table | 决策表**: Logic conditions and actions
- **Decision Tree | 决策树**: Graphical representation of decisions

**Example - Structured English | 示例 - 结构化英语**:
```
Process: Validate_Order
Input: Order
Output: Validated_Order OR Error_Message

IF Order.Total_Amount > Customer.Credit_Limit THEN
    RETURN Error_Message(\"Credit limit exceeded\")
ELSE IF Order.Quantity < 1 THEN
    RETURN Error_Message(\"Invalid quantity\")
ELSE
    Calculate_Discount(Order)
    RETURN Validated_Order
ENDIF
```

#### 1.2.2 Structured Design (SD) | 结构化设计

**Purpose | 目的**: Transform DFD into modular program structure

**Key Tool: Structure Chart (SC) | 关键工具：结构图**

**Structure Chart Elements | 结构图元素**:
```
┌──────────────┐
│   Module     │  Program module/function
│   模块       │  程序模块/函数
└──────┬───────┘
       │
       │  Module call
       │  模块调用
       ▼
┌──────────────┐
│   Submodule  │
│   子模块     │
└──────────────┘

───→   Data coupling (parameter)
       数据耦合（参数）

───→   Control coupling (flag)
- - -→  控制耦合（标志）
```

**Example Structure Chart | 示例结构图**:
```
             ┌─────────────────┐
             │ Order Management│
             │   System        │
             │ 订单管理系统    │
             └────────┬────────┘
                      │
      ┌───────────────┼───────────────┐
      │               │               │
      ▼               ▼               ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Process  │    │ Validate │    │ Generate │
│ Order    │    │ Order    │    │ Invoice  │
│ 处理订单 │    │ 验证订单 │    │ 生成发票 │
└────┬─────┘    └────┬─────┘    └──────────┘
     │               │
     ▼               ▼
  Order Data    Validation Rules
  订单数据      验证规则
```

**Design Principles | 设计原则**:
- **High Cohesion | 高内聚**: Module performs single, well-defined function
- **Low Coupling | 低耦合**: Minimal dependencies between modules
- **Top-Down Design | 自顶向下设计**: Decompose from high-level to low-level
- **Information Hiding | 信息隐藏**: Hide implementation details

#### 1.2.3 Structured Programming (SP) | 结构化编程

**Three Basic Constructs | 三种基本结构**:

**1. Sequence | 顺序结构**
```
Statement 1
Statement 2
Statement 3
```

**2. Selection | 选择结构**
```
IF condition THEN
    Statement A
ELSE
    Statement B
ENDIF
```

**3. Iteration | 迭代结构**
```
WHILE condition DO
    Statement
ENDWHILE
```

**Benefits | 优势**:
- ✅ Eliminates GOTO statements (improves readability)
- ✅ Easier to understand and maintain
- ✅ Facilitates structured testing

**Advantages and Disadvantages of Structured Method | 结构化方法的优缺点**:

| Advantages<br/>优点                                     | Disadvantages<br/>缺点                                          |
| ------------------------------------------------------- | --------------------------------------------------------------- |
| ✅ Clear, systematic approach<br/>清晰、系统的方法       | ❌ Not suitable for complex systems<br/>不适合复杂系统           |
| ✅ Good for well-defined problems<br/>适合明确定义的问题 | ❌ Difficult to model real-world objects<br/>难以建模现实对象    |
| ✅ Easy to learn and apply<br/>易于学习和应用            | ❌ Poor support for reuse<br/>复用支持差                         |
| ✅ Mature tools and techniques<br/>成熟的工具和技术      | ❌ Maintenance challenges as system grows<br/>系统增长时维护困难 |

**中文:**

**结构化方法**是一种强调自顶向下功能分解和面向过程建模的软件开发方法。

**三个组成部分**:
1. **结构化分析(SA)**: 使用数据流图(DFD)建模系统功能
2. **结构化设计(SD)**: 使用结构图将DFD转换为模块化程序结构
3. **结构化编程(SP)**: 使用顺序、选择、迭代三种基本结构

**[Exam Focus | 考试重点]**:
- Draw DFD for given requirements (Context, Level 0, Level 1)
- Identify data flows, processes, and data stores
- Transform DFD to Structure Chart
- Recognize high cohesion and low coupling in structure charts
- Understand difference between data coupling and control coupling

### 1.3 Object-Oriented Method | 面向对象方法

**English:**

**Object-Oriented Method** is a software development approach that models the real world as a collection of interacting objects, each encapsulating data and behavior.

**Core Concepts | 核心概念**:

1. **Object | 对象**: An instance of a class with state (attributes) and behavior (methods)
2. **Class | 类**: A blueprint for creating objects
3. **Encapsulation | 封装**: Bundling data and methods, hiding internal details
4. **Inheritance | 继承**: Creating new classes from existing ones
5. **Polymorphism | 多态**: Objects of different classes respond to same message differently
6. **Abstraction | 抽象**: Focusing on essential characteristics

**OO Principles | 面向对象原则**:

**SOLID Principles | SOLID原则** (covered in detail in Design Patterns document):
- **S**: Single Responsibility Principle
- **O**: Open-Closed Principle
- **L**: Liskov Substitution Principle
- **I**: Interface Segregation Principle
- **D**: Dependency Inversion Principle

#### 1.3.1 Object-Oriented Analysis (OOA) | 面向对象分析

**Purpose | 目的**: Understand and model the problem domain using objects

**Key Activities | 关键活动**:
1. Identify objects and classes
2. Define attributes and behaviors
3. Establish relationships (association, aggregation, composition, inheritance)
4. Model interactions

### 1.4 Agile Methods | 敏捷方法

**English:**

**Agile Methods** are iterative and incremental approaches that emphasize flexibility, collaboration, and rapid delivery of working software.

**Agile Manifesto | 敏捷宣言**:

> **We value | 我们重视**:
> - **Individuals and interactions** over processes and tools
> - **Working software** over comprehensive documentation
> - **Customer collaboration** over contract negotiation
> - **Responding to change** over following a plan

**12 Agile Principles (Key Ones) | 12条敏捷原则（关键）**:
1. Satisfy customers through early and continuous delivery
2. Welcome changing requirements
3. Deliver working software frequently
4. Daily collaboration between business and developers
5. Build projects around motivated individuals
6. Working software is the primary measure of progress
7. Sustainable development pace
8. Continuous attention to technical excellence
9. Simplicity - maximize work not done
10. Self-organizing teams
11. Regular reflection and adaptation

**Popular Agile Frameworks | 流行的敏捷框架**:

#### 1.4.1 Scrum

**Scrum Roles | Scrum角色**:
- **Product Owner | 产品负责人**: Owns product backlog, prioritizes features
- **Scrum Master | Scrum主管**: Facilitates process, removes impediments
- **Development Team | 开发团队**: Cross-functional, self-organizing team (5-9 members)

**Scrum Artifacts | Scrum工件**:
- **Product Backlog | 产品待办列表**: Prioritized list of features
- **Sprint Backlog | 迭代待办列表**: Work committed for current sprint
- **Increment | 增量**: Potentially shippable product increment

**Scrum Events | Scrum事件**:
```
Sprint (1-4 weeks) | 迭代（1-4周）:

Day 1:
  Sprint Planning | 迭代计划会
  - What will be delivered?
  - How will work be achieved?

Daily (15 min):
  Daily Scrum | 每日站会
  - What did I do yesterday?
  - What will I do today?
  - Any impediments?

End of Sprint:
  Sprint Review | 迭代评审
  - Demo to stakeholders
  - Get feedback

  Sprint Retrospective | 迭代回顾
  - What went well?
  - What can be improved?
```

**Scrum Workflow | Scrum工作流**:
```
Product Backlog → Sprint Planning → Sprint Backlog → Daily Scrum → Increment
产品待办       → 迭代计划      → 迭代待办     → 每日站会   → 增量
                                                            ↓
                                                      Sprint Review
                                                      迭代评审
                                                            ↓
                                                      Sprint Retrospective
                                                      迭代回顾
                                                            ↓
                                                      Next Sprint
                                                      下一个迭代
```

#### 1.4.2 Extreme Programming (XP) | 极限编程

**XP Practices | XP实践**:

**Core Practices | 核心实践**:
1. **Test-Driven Development (TDD) | 测试驱动开发**
   ```
   Red → Green → Refactor
   写测试 → 实现 → 重构
   
   1. Write failing test first
   2. Write minimal code to pass
   3. Refactor to improve design
   ```

2. **Pair Programming | 结对编程**
   - Two developers, one computer
   - Driver writes code, Navigator reviews
   - Roles switch frequently

3. **Continuous Integration | 持续集成**
   - Integrate code multiple times per day
   - Automated build and test
   - Immediate feedback on integration issues

4. **Collective Code Ownership | 集体代码所有权**
   - Anyone can modify any code
   - Reduces knowledge silos

5. **Simple Design | 简单设计**
   - Passes all tests
   - No duplication
   - Expresses intent clearly
   - Minimal classes and methods

6. **Refactoring | 重构**
   - Continuous improvement of code design
   - Without changing behavior

**Advantages and Disadvantages of Agile | 敏捷方法的优缺点**:

| Advantages<br/>优点                     | Disadvantages<br/>缺点                             |
| --------------------------------------- | -------------------------------------------------- |
| ✅ Flexible to changes<br/>灵活应对变化  | ❌ Less predictable<br/>可预测性较低                |
| ✅ Faster time-to-market<br/>更快上市    | ❌ Requires customer involvement<br/>需要客户参与   |
| ✅ Higher quality (TDD, CI)<br/>更高质量 | ❌ Not suitable for all projects<br/>不适合所有项目 |
| ✅ Better team morale<br/>更好的团队士气 | ❌ Documentation may be lacking<br/>文档可能不足    |
| ✅ Continuous feedback<br/>持续反馈      | ❌ Requires experienced team<br/>需要经验丰富的团队 |

**中文:**

**敏捷方法**是强调灵活性、协作和快速交付可工作软件的迭代增量方法。

**流行框架**: Scrum、极限编程(XP)、看板(Kanban)

### 1.5 Method Selection Strategy | 方法选择策略

**English:**

**Decision Factors | 决策因素**:

| Factor<br/>因素                           | Structured<br/>结构化 | Object-Oriented<br/>面向对象   | Agile<br/>敏捷         |
| ----------------------------------------- | --------------------- | ------------------------------ | ---------------------- |
| **Requirements Stability<br/>需求稳定性** | Stable<br/>稳定       | Moderately stable<br/>中等稳定 | Volatile<br/>易变      |
| **System Complexity<br/>系统复杂度**      | Low-Medium<br/>低-中  | Medium-High<br/>中-高          | Medium<br/>中          |
| **Team Size<br/>团队规模**                | Large<br/>大          | Medium-Large<br/>中-大         | Small-Medium<br/>小-中 |
| **Project Duration<br/>项目周期**         | Long<br/>长           | Medium-Long<br/>中-长          | Short-Medium<br/>短-中 |
| **Customer Involvement<br/>客户参与**     | Low<br/>低            | Medium<br/>中                  | High<br/>高            |
| **Documentation Need<br/>文档需求**       | High<br/>高           | Medium<br/>中                  | Low<br/>低             |

**Selection Guidelines | 选择指南**:

**Use Structured Method When | 使用结构化方法当**:
- ✅ Requirements are well-defined and stable
- ✅ System is data-processing oriented
- ✅ Large, distributed team
- ✅ Regulatory compliance requires extensive documentation

**Use Object-Oriented Method When | 使用面向对象方法当**:
- ✅ Modeling complex real-world domains
- ✅ Need for reusability and extensibility
- ✅ Long-term maintenance expected
- ✅ System has complex behavior and interactions

**Use Agile Methods When | 使用敏捷方法当**:
- ✅ Requirements are unclear or changing
- ✅ Need frequent releases
- ✅ Customer can be actively involved
- ✅ Small, co-located, experienced team

**Hybrid Approach | 混合方法**:

Many projects use a combination:
- **Agile OO**: Use OO analysis/design with Agile iterations
- **Structured Core, Agile Extensions**: Stable core with structured method, new features with Agile
- **Water-Scrum-Fall**: Requirements and architecture upfront, iterative development, final validation

**中文:**

根据需求稳定性、系统复杂度、团队规模、项目周期、客户参与度和文档需求选择合适的方法。

**[Exam Focus | 考试重点]**:
- Given a project scenario, recommend appropriate development method
- Justify method selection based on project characteristics
- Understand when to use hybrid approaches
- Recognize advantages and disadvantages of each method
